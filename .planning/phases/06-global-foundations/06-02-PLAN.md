<task type="auto">
  <name>Task 1: Add uppercase and text-wrap balance to global heading rule</name>
  <files>frontend/src/app/globals.css</files>
  <action>
In `globals.css`, the existing `h1, h2, h3, h4, h5, h6` rule (around line 60) only sets `font-family`. Extend it to include:

```css
h1, h2, h3, h4, h5, h6 {
  font-family: var(--font-quablo), system-ui, serif;
  text-transform: uppercase;
  text-wrap: balance;
}
```

**Why `text-transform: uppercase` at the CSS level:**
- Many heading translation strings are already uppercase in the JSON files (e.g., "LÍDERES EUROPEOS EN PLATOS ECOLÓGICOS"). Adding CSS uppercase has no visual effect on already-uppercase text — it just ensures consistency for any heading that might use mixed-case.
- Some headings in components use Tailwind `uppercase` class inline. The CSS rule and the Tailwind class are additive but not conflicting (`text-transform: uppercase` is idempotent).
- This makes ALL headings uppercase by default, matching the brand design direction specified in the requirements.

**Why `text-wrap: balance` at the CSS level:**
- `text-wrap: balance` distributes text evenly across lines, preventing orphan words. It is well-supported in modern browsers (Chrome 114+, Firefox 121+, Safari 17.5+).
- Some headings may already have `text-wrap: balance` via Tailwind utility. The CSS property is idempotent — applying it twice has no effect.
- This prevents awkward single-word line breaks on narrow viewports across all current and future headings.

Do NOT remove any existing `uppercase` Tailwind classes from components — they are harmless and removing them is unnecessary churn. The global CSS rule serves as the default; per-component classes remain as documentation of intent.

Do NOT add `letter-spacing` or any other typographic properties in this rule — those vary per heading size and are handled by component-level Tailwind classes.
